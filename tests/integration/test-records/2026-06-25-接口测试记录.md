## 二、认证接口测试

### API-AUTH-001：正确账号登录 ✅

```bash
curl -X POST http://127.0.0.1:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}'
```

**实际响应**：
```json
{
  "code": 200,
  "message": "login success",
  "data": {
    "token": "8e95b7383c8930f2cefe098b8e8270fb",
    "username": "admin",
    "role": "admin"
  }
}
```

**结果**：通过。HTTP 200，token 32位hex，username/role正确。

### API-AUTH-003：无token访问受保护接口 ✅

```bash
curl http://127.0.0.1:8000/api/system/status
```

**实际响应**：
```json
{"detail": {"code": 401, "message": "unauthorized", "detail": "未登录或token无效"}}
```

**结果**：通过。HTTP 401，拒绝无token请求。前端可根据code:401跳转登录页。

---

### API-AUTH-004：空用户名 ⚠️

```bash
curl -X POST http://127.0.0.1:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"","password":"admin123"}'
```

**实际响应**：
```json
{"code": 401, "message": "login failed", "detail": "账号或密码错误"}
```

**预期（按API设计）**：`code:400`，提示"用户名或密码不能为空"

**结果**：行为偏离。后端将空用户名视为"账号错误"而非"参数校验失败"。

**分析**：当前行为不影响功能（空用户名登录必然失败），前端已有表单校验拦截空值，用户不会直接调用API。与陈磊确认：当前行为可接受，前端表单校验已覆盖。

---

### API-AUTH-005：空密码 ⚠️

```bash
curl -X POST http://127.0.0.1:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":""}'
```

**实际响应**：同API-AUTH-004，返回401而非400。

**分析**：同API-AUTH-004。前端表单校验已拦截空值，不影响实际使用。
