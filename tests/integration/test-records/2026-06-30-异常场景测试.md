# 2026-06-30 异常场景测试

## 测试目标

验证系统在异常场景下的容错能力：摄像头设备不可用、token过期、并发操作、数据库异常。

## 测试环境

- 后端: uvicorn, http://localhost:8000
- 数据库: SQLite (system.db)
- 摄像头: 本地笔记本摄像头 (index=0)

## 详细测试记录

### TC-E1: 无效摄像头索引

```
POST /api/camera/start {camera_index: 99, width: 640, height: 480}
→ 503 {
  "detail": "无法打开摄像头 index=99"
}
系统状态正常，不崩溃 ✅
```

### TC-E2-3: 无摄像头时查询/取流

```
# 不启动摄像头直接查询状态
GET /api/camera/status
→ 200 {"status":"offline"}

# 不启动摄像头直接取流
GET /api/camera/stream
→ 流无帧或返回空，不崩溃
```

### TC-E4-6: Token鉴权边界

```
# 无效格式
GET /api/statistics/overview
Authorization: Bearer abc123
→ 401 "无效的访问凭证"

# 空token
GET /api/statistics/overview
Authorization: Bearer
→ 401

# 无Authorization头
GET /api/statistics/overview
→ 401
```

### TC-E7: 空数据库统计

```
# 清空events表后查询统计
GET /api/statistics/overview
→ 200 {
  "today_event_count": 0,
  "total_event_count": 0,
  "recent_events": [],
  "daily_trend": [0,0,0,0,0,0,0]
}
不报错 ✅
```

### TC-E8: 参数并发读写

```
# 连续两次PUT
PUT /api/settings {detection_threshold: 0.45} → 200
PUT /api/settings {detection_threshold: 0.55} → 200
# 验证最终值
GET /api/settings → detection_threshold = 0.55 ✅
```

### TC-E9: 空body参数更新

```
PUT /api/settings {} (空body)
→ 200/422 (取决于实现，不崩溃)
```

### TC-E10: 摄像头快速启停

```
start → 200
stop  → 200
start → 200
stop  → 200
无死锁、无残留线程 ✅
```

### TC-E11: 不存在路由

```
GET /api/nonexistent
→ 404
```

## 数据库验证

```
异常操作后 system.db 完整性检查:
- 无锁文件残留
- WAL文件正常
- 表结构完整
```

## 结论

- 系统对异常输入有合理防护，不会崩溃
- Token鉴权覆盖所有受保护接口
- 摄像头不可用时优雅降级，不影响其他功能
- 参数并发写入无竞态问题（SQLite单写序列化）
