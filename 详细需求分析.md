# 高空抛物监测系统详细需求分析

## 1. 项目背景

高空抛物具有突发性强、危害大、取证困难等特点。传统人工巡查依赖人眼观察，存在漏看、反应慢、证据保存不完整等问题。

本项目面向课程生产实习答辩，目标是完成一个可演示、可测试、可说明的高空抛物监测原型系统。系统重点展示从视频输入、目标检测、轨迹跟踪、行为判断、报警提示到事件存储和回放的完整闭环。

相关文档引用：

- 架构细节见 [docs/architecture.md](docs/architecture.md)。
- 开发环境和启动方式见 [docs/development-guide.md](docs/development-guide.md)。
- GitHub 协作流程见 [GitHub团队协作指南.md](GitHub团队协作指南.md)。
- 排期、交付物、PPT 和完整项目规划见 [高空抛物监测系统-前期规划.md](高空抛物监测系统-前期规划.md)。

## 2. 项目目标

### 2.1 总体目标

构建一个基于 Python、OpenCV、YOLO、DeepSORT、FastAPI 和 Vue 的高空抛物监测系统，实现对视频中疑似坠落物体的检测、跟踪、报警和回放。

### 2.2 答辩目标

- 能在笔记本电脑上稳定运行。
- 能通过上传视频完整演示检测流程。
- 能展示检测框、目标轨迹、报警结果和事件记录。
- 能说明算法原理、系统架构、测试方案和五人分工。
- 能形成项目开发计划、日志、周志、源代码、测试报告和答辩 PPT。

## 3. 用户角色分析

| 用户角色 | 主要需求 |
|---|---|
| 管理员 | 上传视频、查看实时监控、接收报警、查询历史事件、回放视频、配置检测参数、查看系统状态 |

### 3.1 验收方说明

答辩教师不是系统用户，只作为项目验收方，主要关注项目完整性、技术路线、演示效果、测试报告和分工证据。

## 4. 使用场景分析

### 4.1 上传视频检测场景

用户上传本地视频，系统读取视频帧，执行目标检测、目标跟踪和轨迹判断。若发现疑似高空抛物事件，则触发报警，保存截图、轨迹信息和结果视频。

这是本项目的主演示场景。

### 4.2 笔记本摄像头检测场景

系统调用笔记本摄像头，显示实时画面，并尝试对画面中的运动目标进行检测。该场景主要用于证明系统具备实时视频接入能力。

### 4.3 历史事件回放场景

用户进入历史事件页面，按时间、置信度、状态筛选事件，查看事件截图、检测结果和视频片段。

### 4.4 参数配置场景

用户调整检测置信度、连续帧数、最小位移、报警冷却时间等参数，用于适配不同视频场景。

## 5. 功能需求

### 5.1 视频采集与输入

- 支持上传本地视频文件。
- 支持选择预置演示视频。
- 支持调用笔记本摄像头。
- 支持视频格式校验。
- 支持显示视频读取失败、格式错误、文件过大等异常提示。

### 5.2 视频预处理

- 支持视频解码和逐帧读取。
- 支持帧缩放，降低推理压力。
- 支持基础去噪，提高画面稳定性。
- 支持对比度增强，改善低对比度视频效果。
- 支持 ROI 区域选择，只分析楼体或重点区域。

### 5.3 目标检测

- 使用 YOLO 检测视频中的疑似坠落物。
- 在画面上绘制检测框、类别、置信度。
- 支持设置检测置信度阈值。
- 支持 GPU 推理，GPU 不可用时允许 CPU 运行。
- 检测类别统一按“疑似坠落物”处理。

### 5.4 目标跟踪

- 使用 DeepSORT 为目标分配 ID。
- 记录目标连续出现帧数。
- 记录目标中心点轨迹。
- 在画面上绘制运动轨迹。
- 支持目标丢失后的短时间容错。

### 5.5 行为识别

系统根据轨迹规则判断疑似高空抛物事件，核心判断条件包括：

| 判断条件 | 说明 | 建议初始值 |
|---|---|---|
| 运动方向 | 目标中心点整体向下移动 | 连续下降帧占比 ≥ 70% |
| 垂直位移 | 目标在 y 轴方向的累计下降距离 | ≥ 80 像素，可按视频分辨率调整到 50～150 像素 |
| 连续帧数 | 目标被连续检测和跟踪的帧数 | ≥ 5 帧 |
| 检测置信度 | YOLO 输出的目标置信度 | ≥ 0.35 |
| ROI 区域 | 轨迹主要位于监控区域内 | 轨迹点 ≥ 70% 位于 ROI 内 |
| 报警冷却 | 同一疑似事件避免重复报警 | 10 秒 |

本项目不精确区分“人为抛出”和“自然掉落”，统一定义为“疑似高空抛物事件”。

### 5.6 实时报警

- 检测到疑似事件后，页面弹窗报警。
- 支持本地声音报警。
- 支持记录报警时间。
- 支持报警冷却，避免同一事件重复报警。
- 支持事件确认状态。

短信、邮件、企业微信通知不作为本期必做功能，只作为扩展方向。

### 5.7 数据存储

- 保存上传原视频。
- 保存分析后结果视频。
- 保存事件截图。
- 保存事件时间、置信度、轨迹、视频路径和处理状态。
- 保存系统参数配置。
- 使用 SQLite 作为本地数据库。

### 5.8 历史回放

- 支持查看历史报警事件列表。
- 支持按时间筛选事件。
- 支持查看事件详情。
- 支持播放结果视频或事件片段。
- 支持查看事件截图和轨迹信息。

### 5.9 用户界面

前端至少包含以下页面：

- 管理员登录页：使用单一管理员账号登录系统。
- 系统首页：展示系统状态和核心数据。
- 视频分析页：上传视频、选择 ROI、启动分析、查看结果。
- 实时监控页：显示摄像头画面和检测结果。
- 报警中心页：显示报警弹窗和事件列表。
- 历史事件页：查询、筛选、回放事件。
- 数据看板页：展示今日报警数、累计事件数、事件趋势、置信度分布和视频源状态。
- 参数设置页：配置检测阈值、报警参数和视频源。

### 5.10 系统管理

- 支持单一管理员账号登录。
- 支持查看后端服务状态。
- 支持查看算法模块状态。
- 支持查看当前运行设备信息。
- 支持显示 FPS、处理进度和任务状态。
- 支持异常信息提示。

## 6. 非功能需求

### 6.1 可演示性

- 系统必须能在答辩现场稳定启动。
- 上传视频检测流程必须完整可走通。
- 主演示视频应提前准备并本地保存。
- 断网情况下核心功能仍可演示。

### 6.2 实时性

- 上传视频分析应显示处理进度。
- 摄像头检测目标帧率 ≥ 15 FPS；GPU 不足时允许降到 10 FPS，但必须保证画面连续显示。
- GPU 环境下优先保证流畅演示。
- CPU 环境下允许降低分辨率或抽帧。

### 6.3 准确性

- 使用公开数据集和预训练模型提升检测效果。
- 使用 ROI 和轨迹规则降低误报。
- 提供测试集结果，不夸大泛化能力。

### 6.4 稳定性

- 视频读取失败不能导致系统崩溃。
- 模型文件缺失应给出明确提示。
- 数据库异常应返回错误信息。
- 前端刷新后不应影响已保存事件。

### 6.5 易用性

- 页面操作流程清晰。
- 报警信息醒目。
- 参数名称易理解。
- 历史事件可快速定位和回放。

## 8. 数据需求

### 8.1 数据来源

- FADE 公开数据集。
- 阿里云天池高空抛物相关公开数据集。
- 其他公开视频或测试视频。

### 8.2 数据类型

- 原始视频数据。
- 抽帧图片数据。
- 标注数据。
- 模型训练结果。
- 系统事件数据。
- 测试结果数据。

### 8.3 数据保存内容

| 数据 | 保存内容 |
|---|---|
| 视频任务 | 文件名、路径、上传时间、任务状态 |
| 检测结果 | 检测框、置信度、帧号、时间戳 |
| 跟踪结果 | 目标 ID、轨迹点、连续帧数 |

| 系统参数 | 置信度、ROI、位移阈值、报警冷却时间 |

### 8.4 数据建模

本系统数据分为三类：数据库结构化数据、 本地文件数据、算法过程数据。SQLite 负责保存任务、事件和参数；本地目录负责保存原视频、结果视频、截图和模型文件；算法过程数据用于生成检测框、轨迹和事件。

```text

video_tasks 1 —— N tracking_results
```

## 9. 功能建模

### 9.1 登录与权限校验模型

| 项目 | 内容 |
|---|---|
| 输入 | 管理员账号、密码 |
| 输出 | 登录成功 token 或失败提示 |
| 参与模块 | 前端登录页、后端认证接口 |
| 数据 | 固定账号 `admin/admin123` |

正常流程：

```text
管理员打开登录页
→ 输入账号密码
→ 前端提交到后端
→ 后端校验账号密码
→ 校验成功返回 token
→ 前端保存 token
→ 跳转系统首页
```

异常流程：

```text
账号或密码错误
→ 后端返回登录失败
→ 前端提示“账号或密码错误”

token 缺失或失效
→ 后端返回 401
→ 前端跳转登录页
```

### 9.2 上传视频分析功能模型

| 项目 | 内容 |
|---|---|
| 输入 | 视频文件、ROI 参数、检测参数 |
| 输出 | 分析任务、进度、结果视频、报警事件 |
| 参与模块 | 前端视频分析页、后端任务模块、算法模块、数据库、文件存储 |
| 数据 | `video_tasks`、`events`、原视频、结果视频、截图 |

正常流程：

```text
管理员进入视频分析页
→ 选择本地视频
→ 前端读取视频首帧
→ 管理员绘制矩形 ROI
→ 点击开始分析
→ 后端保存原视频到 uploads/
→ 后端读取 total_frames
→ 后端创建 video_tasks，status=pending
→ 后端启动后台线程
→ 任务状态改为 running
→ 算法逐帧读取视频
→ YOLOv11 检测疑似坠落物
→ DeepSORT 生成目标 ID 和轨迹
→ 轨迹规则判断疑似事件
→ 算法保存截图和结果视频
→ 后端写入 events
→ 任务状态改为 success
→ 前端轮询任务进度
→ 前端展示结果视频和事件列表
```

异常流程：

```text
视频格式不支持
→ 后端拒绝上传
→ 前端提示格式错误

视频损坏或无法读取
→ video_tasks.status=failed
→ 写入 error_message
→ 前端显示失败原因

模型文件缺失
→ 算法返回失败
→ 后端记录 error_message
→ 前端提示模型缺失

分析过程中异常中断
→ status=failed
→ 保留原视频和错误日志
```

### 9.3 ROI 选择功能模型

| 项目 | 内容 |
|---|---|
| 输入 | 视频首帧、鼠标拖拽坐标 |
| 输出 | `roi_x`、`roi_y`、`roi_width`、`roi_height` |
| 参与模块 | 前端视频分析页、后端任务模块、算法模块 |
| 数据 | `video_tasks` 中的 ROI 字段 |

正常流程：

```text
前端显示视频首帧
→ 管理员拖拽矩形区域
→ 前端计算 ROI 坐标
→ 创建任务时提交 ROI
→ 后端保存 ROI
→ 算法只对 ROI 内轨迹做事件判断
→ 结果视频绘制 ROI 边框
```

异常流程：

```text
管理员未绘制 ROI
→ 默认使用整帧作为 ROI

ROI 超出画面范围
→ 前端限制坐标
→ 后端再次校验
```

### 9.4 算法检测与跟踪功能模型

| 项目 | 内容 |
|---|---|
| 输入 | 视频路径、ROI、检测阈值 |
| 输出 | 检测框、目标 ID、轨迹、事件、截图、结果视频 |
| 参与模块 | OpenCV、YOLOv11、DeepSORT、轨迹规则模块 |
| 数据 | `detection_results`、`tracking_results`、`events` |

正常流程：

```text
算法接收视频任务
→ OpenCV 打开视频
→ 逐帧读取
→ 图像预处理
→ YOLOv11 输出 bbox 和 confidence
→ DeepSORT 根据 bbox 分配 track_id
→ 保存每个目标中心点
→ 判断目标是否持续向下
→ 判断垂直位移是否达标
→ 判断连续帧数是否达标
→ 判断轨迹点是否主要在 ROI 内
→ 命中规则后生成事件
→ 绘制检测框、ID、轨迹、ROI
→ 输出结果视频
```

异常流程：

```text
视频帧读取失败
→ 跳过异常帧或终止任务

某一帧无检测结果
→ DeepSORT 维持短时间目标状态

目标短暂丢失
→ 在容错帧数内继续跟踪

置信度低于阈值
→ 不进入跟踪和事件判断
```

### 9.5 报警事件处理功能模型

| 项目 | 内容 |
|---|---|
| 输入 | 算法生成的疑似事件 |
| 输出 | 报警弹窗、声音提示、事件记录 |
| 参与模块 | 算法模块、后端事件模块、前端报警中心、数据库 |
| 数据 | `events` |

正常流程：

```text
算法判定疑似事件
→ 保存事件截图
→ 后端写入 events，status=unconfirmed
→ 前端轮询或刷新事件列表
→ 报警中心弹窗
→ 播放本地报警声音
→ 管理员查看截图和结果视频
→ 管理员点击确认或标记误报
→ 后端更新事件 status
→ 历史事件和数据看板同步更新
```

异常流程：

```text
同一事件短时间重复触发
→ 使用 alarm_cooldown_seconds 冷却
→ 不重复写入事件

截图保存失败
→ 事件仍入库
→ snapshot_path 为空
→ 前端提示截图缺失

结果视频未生成
→ 事件仍显示
→ 回放按钮置灰
```

## 10. 接口需求

### 10.1 前后端接口

- 登录接口：校验固定管理员账号，返回简单 token。
- 健康检查接口：返回后端运行状态。
- 系统状态接口：返回后端、算法、数据库状态。
- 视频上传接口：上传视频并创建分析任务。
- 任务查询接口：查询分析进度和结果。
- 事件查询接口：获取历史报警事件。
- 事件状态更新接口：确认事件或标记误报。
- 统计接口：返回今日报警数、累计事件数、趋势、置信度分布和状态分布。
- 参数配置接口：读取和修改系统参数。
- 文件访问接口：访问截图、结果视频和事件片段。
- 摄像头启动接口：打开本机摄像头。
- 摄像头停止接口：释放本机摄像头。
- 摄像头画面接口：使用 MJPEG 流返回实时检测画面，前端通过 `<img>` 或 video 容器展示。

说明：上传视频与创建任务合并为一个接口 `POST /api/videos/upload`，该接口完成保存视频、创建 `video_tasks`、启动后台线程。前端通过 `GET /api/tasks/{id}` 轮询进度。

## 13. 约束条件

- 项目周期短，截止日期为 2026 年 7 月 3 日。
- 无嵌入式设备和外接摄像头。
- 主部署环境为 Windows 笔记本。
- 主训练设备：石义焌笔记本，RTX4060 8GB。
- 主演示设备：罗龙飞笔记本，RTX3050 4GB。
- 备用演示设备：陈磊笔记本，RTX3050 4GB。
- 另外两台无独立 GPU 设备用于 CPU 兼容性验证和文档工作。
- 小组成员 Git 基础较弱，需要严格按协作规范执行。
- 模型训练时间有限，不从零训练大模型。
- 答辩优先级高于真实工程部署。

## 20. 结论

本项目需求重点不是实现工业级高空抛物平台，而是在课程周期内完成一个功能闭环清晰、技术路线合理、测试证据完整、答辩效果稳定的人工智能原型系统。

系统核心价值体现在：能用 AI 检测视频中的疑似坠落物，能跟踪轨迹，能产生报警，能保存证据，能回放历史事件。
