# 6/26 联调记录：上传→任务→算法占位流程

## 一、联调目标

验证 6/26 阶段目标：上传视频后能创建任务并调用算法占位模块。

## 二、流程确认

```
前端上传视频 → POST /api/videos/upload → 后端保存视频+创建任务
                                           ↓
前端绘制ROI  → POST /api/tasks/{id}/analyze → 后端校验状态+启动后台线程
                                                  ↓
                                          线程调用 AnalysisPipeline.analyze()
                                                  ↓
                                          当前返回 status="not_ready"
                                                  ↓
                                          后端标记任务 failed，写入error_message
```

## 三、接口对齐确认

### 3.1 后端 → 前端 (陈磊 ← → 刘康)

| 接口 | 方向 | 字段 | 状态 |
|---|---|---|---|
| POST /api/videos/upload | 后端提供 | file(multipart) → {task_id, filename, total_frames} | ✅ 对齐 |
| POST /api/tasks/{id}/analyze | 后端提供 | {roi_x, roi_y, roi_width, roi_height} → {task_id, status} | ✅ 对齐 |
| GET /api/tasks/{id} | 后端提供 | → {id, status, progress, total_frames, events[]} | ✅ 对齐 |

### 3.2 后端 ← → 算法 (陈磊 ← → 石义焌)

| 调用 | 当前状态 |
|---|---|
| `AnalysisPipeline.analyze(source, roi, detection_params)` | 占位返回 `status="not_ready"` |
| `PipelineResult` 字段 | status, source, message, detections[], tracks[], events[] |
| 6/28 目标 | 替换为 YOLOv11+DeepSORT 真实推理 |

**注意**: 当前 `PipelineResult` 缺少 `result_video_path`、`total_frames`、`processed_frames` 字段。task_service 使用 `getattr` 防御性访问，等算法就绪后补充。
