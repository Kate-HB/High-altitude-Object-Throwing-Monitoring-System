## 三、算法Pipeline接口（严格遵循 docs/algorithm-interface.md）

### 3.1 调用方式

```python
from algorithm.pipeline import run_video_analysis

result: PipelineResult = run_video_analysis(
    video_path="uploads/demo.mp4",
    output_dir="outputs/task_1/",
    roi={"x": 100, "y": 50, "width": 400, "height": 300},
    settings={
        "detect_confidence": 0.35,       # YOLO检测阈值
        "downward_ratio": 0.7,            # 下降帧占比阈值
        "min_vertical_distance": 80,      # 最小垂直位移(px)
        "min_track_frames": 5,            # 最小跟踪帧数
        "roi_required_ratio": 0.7,        # ROI内轨迹点占比阈值
        "alarm_cooldown_seconds": 10,     # 同track_id报警冷却(秒)
    },
)
```

### 3.2 返回值：PipelineResult dataclass

```python
@dataclass
class PipelineResult:
    status: str                        # "success" | "failed"
    total_frames: int                  # 视频总帧数
    processed_frames: int              # 实际处理帧数
    result_video_path: str | None      # 结果视频路径
    events: list[dict]                 # EventInfo列表（见3.3）
    detection_results: list[dict]      # DetectionInfo列表（见3.4）
    tracking_results: list[dict]       # TrackingInfo列表（见3.5）
    error_message: str | None          # 失败原因
```

### 3.3 EventInfo 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| track_id | int | DeepSORT 目标ID |
| confidence | float | 事件置信度 |
| snapshot_path | str | 报警瞬间截图路径 |
| created_at | str | 事件时间，格式 `YYYY-MM-DD HH:mm:ss` |

> 注意：不含 frame_id 和 bbox。陈磊 `_write_events()` 按此四字段入库。

### 3.4 DetectionInfo 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| frame_id | int | 帧编号 |
| bbox_x / bbox_y / bbox_width / bbox_height | float | 检测框 |
| confidence | float | 检测置信度 |
| class_name | str | 固定 `"falling_object"` |

### 3.5 TrackingInfo 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| track_id | int | 目标ID |
| frame_id | int | 帧编号 |
| timestamp | float | 帧时间戳（秒） |
| center_x / center_y | float | 中心点 |
| bbox_x / bbox_y / bbox_width / bbox_height | float | 跟踪框 |

### 3.6 处理流程

```
视频输入 → 逐帧YOLO检测 → detect_confidence过滤 → DeepSORT跟踪(track_id) → 轨迹累积
                                                                                  ↓
结果视频 ← 绘制检测框+轨迹+ROI ← 行为判断(5条件+冷却) → events列表
```

### 3.7 行为识别规则（6参数，来自 system_settings 表）

| 条件 | 参数 | 默认值 | 说明 |
|---|---|---|---|
| YOLO检测置信度 | detect_confidence | ≥0.35 | 低于此值的检测框丢弃 |
| 下降帧占比 | downward_ratio | ≥70% | 目标中心点y坐标连续下降 |
| 最小垂直位移 | min_vertical_distance | ≥80px | 累计垂直位移 |
| 最小跟踪帧数 | min_track_frames | ≥5帧 | 短于此帧数不触发 |
| ROI内轨迹点占比 | roi_required_ratio | ≥70% | 轨迹点落在ROI内的比例 |
| 报警冷却时间 | alarm_cooldown_seconds | ≥10秒 | 同track_id在冷却时间内不重复报警 |

六条件同时满足 → 触发报警事件，保存截图，写入created_at（ISO格式 `YYYY-MM-DD HH:mm:ss`）。

### 3.8 DeepSORT备用方案

若DeepSORT接入阻塞，先用简化方案保证主链路联调：
- 基于IOU的帧间匹配分配临时track_id
- 保留 `pipeline.py` 接口签名不变
- 简化方案与DeepSORT输出格式一致（TrackingInfo字段对齐）
- DeepSORT就绪后替换 `algorithm/tracking/tracker.py` 实现，不影响调用方

## 八、风险与备用方案

| 风险 | 应对 |
|---|---|
| 公开数据集质量差/数量少 | 自行标注2-3段演示视频关键帧（≥100张），用预训练权重+少量微调 |
| YOLOv11训练效果差 | 使用预训练模型直接推理 + 精选演示视频 + 重点靠轨迹规则判断 |
| DeepSORT接入超时 | 保留接口，先用IOU简化track_id跑通主链路（6/28前），后续替换 |
| ONNX导出失败 | PPT中作为尝试优化方向记录，系统继续用PyTorch权重 |
| RTX4060 8GB显存不足（训练） | 降低imgsz（640→480）、减小batch size、启用混合精度 |
| RTX3050 4GB显存不足（推理） | imgsz≤480、batch=1、跳帧推理；主演示视频≤30秒 |
| Pipeline返回字段与合同不一致 | 严格对照 `docs/algorithm-interface.md` 第4节；6/26起每次联调前核对；变更必须更新合同文档 |
| 摄像头推理帧率低 | 摄像头为辅演示，降低分辨率+跳帧推理 |
| 报警冷却时间未实现 | 同track_id重复报警，看板统计失真；6/28起必须实现 |
| 分支未建立/日志缺失 | 6/24当天创建分支+首篇日志，补齐到与其余四人同步 |
