# 1. 项目概述

## 1.1 项目背景

高空抛物被称为"悬在城市上空的痛"。每年全国发生数千起高空抛物事故，造成人员伤亡和财产损失。传统监控存在三大痛点：

| 痛点 | 现状 | 后果 |
|------|------|------|
| 人工查看效率低 | 一人盯几十路摄像头 | 无法实时发现抛物事件 |
| 事后回溯为主 | 出事才翻录像，证据分散 | 错过最佳处理时机 |
| 缺乏实时报警 | 无自动检测+通知能力 | "看到的时候已经晚了" |

## 1.2 项目目标

打造一个 **视频输入 → 目标检测与跟踪 → 行为判定 → 实时报警 → 事件存储与回放** 的全自动闭环 AI 原型系统，面向生产实习答辩交付。

### 1.2.1 主演示闭环（P0，必须完整）

```
登录 → 上传视频 → ROI选择 → YOLOv11检测 → IOU跟踪 → 轨迹判断 → 报警事件 → 数据库入库 → 历史回放 → 数据看板
```

### 1.2.2 辅演示闭环（P1）

```
摄像头实时画面 + YOLOv11检测框标注（不要求报警入库）
```

## 3.2 数据流

```
[视频文件] → 上传 → uploads/ → 创建 video_task (status=pending)
    │
    ▼
POST /api/videos/{id}/analyze → create_task() + _run_analysis() 后台线程
    │
    ├─ ① video_task.status = "running"
    ├─ ② subprocess.run(pipeline_cli.py ...)
    │       ├─ YOLOv11 ONNX 逐帧检测
    │       ├─ ROI 过滤
    │       ├─ IOU 贪心匹配跟踪
    │       ├─ 6条件行为判断 → 触发事件
    │       ├─ 绘制检测框+轨迹+ROI → result.mp4
    │       ├─ 事件帧快照 → snapshot.jpg
    │       └─ 输出 progress.json + results.json
    ├─ ③ 解析JSON → batch_insert_events/detections/tracks
    └─ ④ video_task.status = "success", result_video_path 写入
    │
    ▼
[前端] 轮询 GET /api/videos/{id} → 进度更新
       GET /api/events → 报警列表
       GET /api/statistics/overview → 看板数据
       GET /api/files → 快照/视频
```

| 决策 | 选择 | 替代方案 | 原因 |

### 5.1.2 核心服务说明

| `settings_service.py` | 参数读写+校验 | `get_settings()`, `update_settings()` |
| `video_service.py` | 视频文件管理 | `save_video()`, `get_video_path()` |

### 5.1.3 统一响应格式

```json
{
  "code": 0,        // 0=成功, 非0=错误
  "message": "ok",  // 提示信息
  "data": {}        // 业务数据
}
```

错误码：
- `401`：未登录/Token过期
- `403`：文件路径越权
- `404`：资源不存在
- `409`：资源冲突（如摄像头被占用）
- `413`：文件过大
- `429`：请求频率限制
- `500`：服务器内部错误

# 8. 算法设计

## 8.1 Pipeline 总流程

```
run_video_analysis(video_path, roi, settings)
    │
    ├─ ① 初始化
    │    ├─ YOLODetector(settings.imgsz, settings.detect_confidence)
    │    ├─ IOUTracker()
    │    └─ BehaviorAnalyzer(settings)
    │
    ├─ ② 逐帧处理 (for each frame):
    │    ├─ detector.predict_frame(frame)
    │    │    └─ 输出: [{frame_id, bbox, confidence, class_name}]
    │    ├─ ROI过滤: 保留中心点在ROI内的检测
    │    ├─ tracker.update(detections)
    │    │    └─ 输出: [{frame_id, track_id, bbox, center, status}]
    │    ├─ behavior.analyze(tracks)
    │    │    └─ 6条件联合判断 → EventInfo 或 None
    │    ├─ 绘制标注: 检测框(绿)+轨迹(线)+ROI(蓝)+事件(红)
    │    └─ 写入 progress.json
    │
    ├─ ③ 输出
    │    ├─ result.mp4 (H.264编码)
    │    ├─ snapshots/*.jpg (事件帧)
    │    ├─ results.json (检测+跟踪+事件摘要)
    │    └─ progress.json (0.0→1.0)
    │
    └─ ④ 返回
         └─ results_json → 后端解析 → 入库
```

## 8.2 目标检测（detection/detector.py）

```
class YOLODetector:
    __init__(imgsz=960, conf_threshold=0.35)
        └─ 加载 ONNX 模型 (models/best.onnx)
        └─ 创建 onnxruntime.InferenceSession (GPU)

    predict_frame(frame: np.ndarray) → List[Detection]
        ① BGR→RGB, resize(imgsz, imgsz), /255归一化
        ② ONNX推理: session.run([outputs], {input: tensor})
        ③ 解析输出: [batch, num_det, 6] → [x1,y1,x2,y2,conf,cls]
        ④ NMS过滤
        ⑤ 坐标还原到原始frame尺寸
        ⑥ 返回: [{frame_id, bbox_x, bbox_y, bbox_width, bbox_height, confidence, class_name}]

模型信息:
    - 文件: models/best.onnx (10.3MB)
    - opset: 17
    - imgsz: 960
    - batch: 1 (fixed)
    - 推理速度: ~24ms/帧 (RTX 3050)
    - 显存占用: ~1.9GB
```

## 8.3 目标跟踪（tracking/tracker.py）

```
class IOUTracker:
    __init__()
        └─ tracks: Dict[track_id, TrackInfo]
        └─ next_id: int = 1

    update(detections: List[Detection]) → List[TrackingResult]
        ① 计算当前帧detections与活跃tracks的IOU矩阵
        ② 贪心匹配: 最高IOU > 阈值 → 关联
        ③ 未匹配detection → 新建track (status=tentative)
        ④ 未匹配track → 标记missed
        ⑤ tentative连续N帧confirmed → status=confirmed
        ⑥ missed超过M帧 → status=deleted
        ⑦ 返回: [{frame_id, track_id, bbox, center, confidence, status}]

TrackInfo:
    - track_id: int
    - status: "tentative" | "confirmed" | "deleted"
    - trajectory: List[{frame_id, time, x, y}]
    - first_seen: timestamp
    - last_seen: timestamp
    - missed_count: int

参数:
    - iou_threshold: 0.3
    - tentative_confirm_frames: 3
    - max_missed_frames: 30
```

## 8.4 行为判定（behavior/behavior.py）

### 6条件联合判断

```
analyze(track: TrackInfo, roi: Rect, settings: Settings) → EventInfo | None

条件① 轨迹足够长
    len(track.trajectory) >= settings.min_track_frames  (默认15帧)
    └─ 过滤瞬时噪点和碎片检测

条件② 整体向下运动 ⭐核心判断
    对轨迹点 (frame_id, y) 做最小二乘线性回归
    斜率 >= settings.downward_ratio  (默认0.55)
    └─ 斜率正=向下，斜率负=向上
    └─ 线性回归比简单dy/dt更鲁棒，抗噪声

条件③ 累计垂直位移
    (trajectory[0].y - trajectory[-1].y) >= settings.min_vertical_distance  (默认50px)
    └─ 排除微小晃动

条件④ ROI内轨迹占比
    count(轨迹点在ROI内) / len(trajectory) >= settings.roi_required_ratio  (默认0.5)
    └─ 确保物体在检测区域内活动

条件⑤ 报警冷却
    now - track.last_alarm_time >= settings.alarm_cooldown_seconds  (默认3s)
    └─ 防止同一物体重复报警

条件⑥ 检测置信度
    已在YOLO阶段过滤 (confidence >= detect_confidence)
    此处不再重复判断

全部满足 → 触发 EventInfo:
    - track_id: int
    - timestamp: str
    - confidence: float (关联检测的平均置信度)
    - trajectory: [{frame_id, time, x, y}]
    - snapshot_frame: int (触发帧)

任一不满足 → 返回 None
```

### 三档阈值预设

| 参数 | default | sensitive | strict |
|------|---------|-----------|--------|
| detect_confidence | 0.35 | 0.25 | 0.50 |
| downward_ratio | 0.55 | 0.45 | 0.70 |
| min_track_frames | 15 | 10 | 25 |
| min_vertical_distance | 50 | 30 | 80 |
| roi_required_ratio | 0.5 | 0.3 | 0.7 |
| **检出率** | 平衡 | 高 | 低 |
| **误报率** | 低 | 中 | 极低 |
| **适用场景** | **答辩推荐** | 灵敏度演示 | 严格演示 |

## 8.5 结果可视化

```
draw_annotations(frame, detections, tracks, roi, events)
    ├─ 检测框: 绿色矩形 + 类别名 + 置信度
    ├─ 轨迹线: 蓝色折线 (最近N帧track中心点连线)
    ├─ track_id: 白色文字标注
    ├─ ROI: 蓝色虚线矩形
    ├─ 事件帧: 红色边框高亮 + "EVENT"标记
    └─ 输出: H.264编码 (mp4v codec → .mp4)
```

## 8.6 子进程桥接

```
后端 (.venv, CPU)  →  subprocess.run()  →  pipeline_cli.py (Conda, GPU)

pipeline_cli.py:
    ① 解析命令行参数: --video, --roi, --settings
    ② 调用 run_video_analysis(...)
    ③ 输出 results.json 到 STDOUT
    ④ 后端捕获 STDOUT → json.loads() → 入库

为什么用子进程?
    - 后端.venv 没有 ultralytics (CPU环境)
    - Conda环境有 torch + ultralytics (GPU)
    - 子进程隔离，环境互不污染
```

# 15. 协作规范

## 15.1 GitHub工作流

### 分支策略

```
main  ───────────────────────────── ● (答辩后合并)
         \
dev       ─────●────●────●────●────● (每日集成)
           \     \    \    \    \
feature/   户名-模块  ...  ...  ...  ... (个人开发)
```

| 分支 | 用途 | 保护 |
|------|------|------|
| `main` | 稳定版本，答辩后合并 | 禁止直接push |
| `dev` | 每日集成，候选版本 | 禁止直接push，需PR |
| `feature/<姓名>-<模块>` | 个人开发分支 | 个人自由push |

### 提交流程

```
1. 个人分支开发 + git add/commit
2. git push origin feature/<姓名>-<模块>
3. GitHub 创建 PR → dev
4. 模块负责人review + 罗龙飞最终审核
5. 通过后合并到dev
6. 每2天端到端集成一次
```

### Commit规范

```
格式: <类型>: <简短描述>

类型:
  feat:     新功能
  fix:      修复Bug
  test:     测试相关
  docs:     文档变更
  chore:    工程/工具变更
  refactor: 重构

示例:
  feat: 实现视频上传接口
  fix: 修复Token过期校验
  test: 新增登录页E2E测试
  docs: 更新API设计文档
```

### PR规范

PR必须包含：
- **改了什么**：变更的文件和功能
- **怎么测试**：手动测试步骤或自动化测试结果
- **是否影响别人**：是否改了共享接口/表结构/依赖

禁止：
- 直接push到main或dev
- 跨模块修改未提前说明
- PR不写描述直接合并

## 15.2 代码规范

- 遵循CLAUDE.md中的编码规范
- 匹配现有代码风格，不引入新的代码风格
- 最小化变更，不"顺手"改无关代码
- 后端：Python类型注解 + pydantic校验
- 前端：Vue 3 Composition API + `<script setup>`

## 15.3 文档规范

### 个人日志格式

```markdown
# 日期 工作日志

## 今日任务
## 完成结果
## 遇到的问题
## 解决办法
## 明日计划
## 相关截图或代码提交
```

重点：**写问题和解决办法**，不是流水账。

### 个人日志提交

每人每天一篇，放到 `docs/logs/` 目录，文件名格式：`YYYY-MM-DD-姓名.md`
