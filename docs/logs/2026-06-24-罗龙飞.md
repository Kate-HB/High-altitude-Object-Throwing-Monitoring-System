## 遇到的问题

### 问题1：需求分析一开始偏宏观

最初需求分析只描述了功能模块，但缺少模块之间的输入、输出和数据流关系。例如“数据看板”只说要展示统计结果，但没有说明统计数据来自哪里、如何计算、和事件表如何关联。

### 问题2：文档之间存在口径不一致

需求分析、前期规划、开发计划之间出现过以下不一致：

- DeepSORT 是否确定使用。
- 是否需要登录功能。
- 数据看板是否属于必须页面。
- ONNX Runtime 是低优先级还是答辩亮点。
- 数据看板负责人是否包含陈磊。
- 6月27日是否安排开发。

### 问题3：接口清单不够细

需求分析中已有接口范围，但只能说明“有哪些接口”，不能直接指导前后端联调。缺少 URL、请求参数、响应结构、错误码和测试点。

### 问题4：ROI流程和上传流程存在冲突

原 API 设计中只允许上传接口同时提交 ROI，但开发计划中前端需要先上传视频、显示首帧，再绘制 ROI。如果不补充启动分析接口，前后端流程会不一致。

### 问题5：团队 Git 基础较弱

小组成员对 Git 的理解主要停留在上传和拉取，容易出现直接改主分支、分支不同步、冲突不会处理等问题。

## 解决办法

- 将需求分析进一步细化为数据建模、功能建模、接口需求、页面接口数据关系和模块分工关系。
- 统一系统口径：
  - 本期只设置管理员一个系统角色。
  - 使用 YOLOv11 做目标检测。
  - 使用 DeepSORT 做目标跟踪。
  - SQLite 作为本地数据库。
  - 摄像头只作为辅演示，不要求报警入库。
  - 管理员账号固定为 `admin/admin123`。
- 将“接口范围清单”进一步扩展为《详细API设计》，明确每个接口的请求、响应、错误和测试点。
- 新增 `POST /api/tasks/{task_id}/analyze`，解决上传后绘制 ROI 再启动分析的问题。
- 在开发计划中要求每天 18:00 做接口同步，避免前端、后端、算法字段不一致。
- 明确 GitHub 协作规则：
  - `main` 只放稳定版本。
  - `dev` 用于每日集成。
  - 个人功能使用 `feature/姓名-模块` 分支。
  - 禁止直接改 `main`。
  - 合并前必须自测并说明影响范围。
